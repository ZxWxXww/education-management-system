# 管理员角色权限管理页面设计

## 1. 背景

当前 [RolePermission.vue](/D:/education/frontend/src/views/admin/RolePermission.vue) 仍是空壳页，但后端已经具备角色分页、详情、创建、更新、删除、权限分配接口。缺口主要在两个方面：

1. 缺少一个可供前端构建权限树的“权限目录接口”。
2. 前端没有真正的角色权限管理页面，导致角色与权限只能停留在接口层，无法完成联调验收。

本轮目标是在不推翻现有 RBAC 架构的前提下，补齐一套真实可用的“角色表格 + 角色编辑抽屉 + 权限树抽屉”页面。

## 2. 目标与范围

### 2.1 目标

实现管理员端“角色权限管理”页面，满足以下能力：

1. 角色列表查询。
2. 角色新增。
3. 角色编辑。
4. 角色删除。
5. 角色权限分配与保存。
6. 保存后刷新页面数据，并明确提示“对该角色用户在下次重新登录后生效”。

### 2.2 非目标

本轮不做以下内容：

1. 不强制让已登录用户即时失效。
2. 不做服务端 token 黑名单或会话踢出。
3. 不重构现有登录态模型。
4. 不改现有管理员/教师/学生菜单结构。

## 3. 业务约束

### 3.1 内置角色保护

内置角色 `ADMIN`、`TEACHER`、`STUDENT` 视为系统基线角色，必须同时在前后端受保护：

1. 不允许删除。
2. 不允许修改 `roleCode`。
3. 允许修改 `roleName`、`description`、`status`、权限集合。

前端只做交互层禁用，后端必须做最终兜底校验，避免被绕过。

### 3.2 权限生效规则

权限保存后：

1. 当前正在登录的其他用户不强制退出。
2. 被修改角色的用户，在“下次重新登录”后获取新的 JWT 和权限集。
3. 页面成功提示中必须明确这一点，避免管理员误以为是即时广播生效。

## 4. 方案选型

采用“现有角色接口 + 新增权限目录接口 + 双抽屉页面”的方案。

### 4.1 页面结构

页面由三部分构成：

1. 顶部统计区：角色总数、启用数、内置角色数、自定义角色数。
2. 角色表格区：支持关键字查询、刷新、新增、编辑、删除、分配权限。
3. 双抽屉：
   - 角色编辑抽屉：维护角色基础信息。
   - 权限树抽屉：维护角色权限勾选。

### 4.2 选择原因

1. 后端已具备角色 CRUD 和权限保存接口，新增一条权限目录接口即可完成闭环。
2. 权限树与角色基础信息拆开后，交互更清晰，便于把“先建角色、再配权限”和“编辑角色、再调权限”统一处理。
3. 双抽屉方案对现有项目侵入最小，适合当前联调阶段。

## 5. 后端设计

### 5.1 现有接口沿用

保留现有 `AdminRoleController` 契约：

1. `GET /api/admin/roles/page`
2. `GET /api/admin/roles/{id}`
3. `POST /api/admin/roles`
4. `PUT /api/admin/roles/{id}`
5. `DELETE /api/admin/roles/{id}`
6. `PUT /api/admin/roles/{id}/permissions`

### 5.2 新增权限目录接口

新增：

1. `GET /api/admin/roles/permissions/catalog`

用途：

1. 返回所有可分配权限的目录数据。
2. 为前端构建树形权限面板提供稳定契约。

建议返回结构：

```json
[
  {
    "groupCode": "ADMIN",
    "groupName": "管理员端",
    "children": [
      {
        "id": 1,
        "permCode": "dashboard:view",
        "permName": "查看仪表盘",
        "resourcePath": "/admin/dashboard",
        "httpMethod": "GET"
      }
    ]
  }
]
```

### 5.3 权限分组规则

权限目录按 `permCode` 前缀归组：

1. `teacher:` 开头归入 `TEACHER`。
2. `student:` 开头归入 `STUDENT`。
3. 其余权限默认归入 `ADMIN`。

这样能兼容当前 `seed-rbac.sql` 的权限模型，同时避免额外维护一张“权限模块配置表”。

### 5.4 后端保护逻辑

在 `AdminRoleServiceImpl` 中新增系统角色保护规则：

1. 删除角色时，如果 `roleCode` 是 `ADMIN / TEACHER / STUDENT`，拒绝删除。
2. 更新角色时，如果目标角色是系统角色，忽略或拒绝 `roleCode` 变更。
3. 权限分配允许作用于系统角色。

### 5.5 错误语义

后端错误信息需明确可读：

1. “系统内置角色不允许删除”
2. “系统内置角色不允许修改角色编码”
3. “角色不存在”
4. “权限列表不能为空”

## 6. 前端设计

### 6.1 API 层

在 [roleAuth.js](/D:/education/frontend/src/api/admin/roleAuth.js) 所在的 admin API 体系中补齐角色权限管理所需封装：

1. 获取角色分页列表。
2. 获取角色详情。
3. 新增角色。
4. 编辑角色。
5. 删除角色。
6. 获取权限目录。
7. 保存角色权限。

其中：

1. 现有“用户角色授权”逻辑继续保留在该文件或相邻 admin 模块中。
2. 新增角色权限页 API 命名要与页面语义一致，避免把“角色管理”和“用户授权”继续混在一起。

### 6.2 页面交互

角色表格字段至少包括：

1. `roleCode`
2. `roleName`
3. `description`
4. `status`
5. 权限数量
6. 权限摘要
7. 操作列

操作列包括：

1. 编辑
2. 分配权限
3. 删除

系统角色表现：

1. 删除按钮禁用。
2. 编辑抽屉中的 `roleCode` 输入框只读。
3. 行内展示“系统内置”标识。

### 6.3 角色编辑抽屉

角色编辑抽屉字段：

1. `roleCode`
2. `roleName`
3. `description`
4. `status`

行为规则：

1. 新增角色时，`roleCode` 可编辑。
2. 编辑系统角色时，`roleCode` 只读。
3. 保存成功后刷新角色表格。
4. 新增成功后允许继续点击“分配权限”。

### 6.4 权限树抽屉

权限树按三个顶层模块展示：

1. `管理员端`
2. `教师端`
3. `学生端`

每个模块下按权限项展示复选节点，支持：

1. 全选
2. 半选
3. 回显当前已选权限
4. 清空
5. 恢复到当前角色已有权限

节点展示文本为：

1. 主文本：`permName`
2. 次信息：`permCode`

这样管理员能同时看到中文语义和真实权限码。

### 6.5 数据流

页面初始化时并行获取：

1. 角色分页列表
2. 权限目录

角色编辑流：

1. 点击新增或编辑
2. 打开角色编辑抽屉
3. 提交基础信息
4. 保存成功后刷新列表

权限分配流：

1. 点击“分配权限”
2. 打开权限树抽屉
3. 以当前角色 `permissionIds` 回显勾选状态
4. 点击保存
5. 调用 `PUT /api/admin/roles/{id}/permissions`
6. 成功后刷新角色列表和该角色详情摘要
7. 显示“下次重新登录后生效”的提示

## 7. 登录态与权限刷新策略

本轮不做服务端强制刷新已登录用户权限。

因此：

1. 修改某角色权限后，当前管理员自身登录态不自动重签 JWT。
2. 被修改角色的用户需重新登录，才能拿到新的 `roles` 和 `permissions`。
3. 菜单显示与接口权限变化，依赖现有登录逻辑重新下发的 `LoginResponse.permissions`。

当前认证链路已经满足这一点：

1. 登录时由后端按激活角色重新加载权限。
2. JWT 中写入当前角色和权限码集合。
3. 前端菜单和路由守卫依据 `permissions` 实时过滤。

## 8. 测试设计

### 8.1 前端契约测试

新增或扩展前端测试，锁定以下行为：

1. `RolePermission.vue` 不再是空壳页。
2. 页面存在角色列表、角色编辑、权限树相关交互文案。
3. API 层存在真实角色管理与权限目录请求封装。
4. 权限目录请求路径能在后端控制器找到对应映射。

### 8.2 后端验证

至少验证：

1. 后端可编译通过。
2. `GET /api/admin/roles/permissions/catalog` 契约稳定可用。
3. 系统角色删除和改码保护生效。

### 8.3 联调验收

手工验收路径：

1. 管理员进入“角色权限管理”页面。
2. 新增一个角色。
3. 打开权限树，勾选若干权限并保存。
4. 到“用户角色授权”页面，把该角色分配给一个测试账号。
5. 使用该账号重新登录。
6. 验证菜单可见性与受保护接口访问结果符合新权限。

## 9. 风险与处理

### 9.1 权限分组并不总是等于页面模块

风险：

1. 当前权限码存在一部分“跨端通用权限”，例如 `resource:view`、`assignment:manage`。

处理：

1. 先按 `teacher:` / `student:` 前缀规则归组，其余归入管理员组。
2. 页面文案中明确这是“系统权限目录”，不是严格的页面导航树。

### 9.2 系统角色保护如果只做前端禁用会被绕过

处理：

1. 必须同时在后端 service 层拦截删除和改码。

### 9.3 页面保存后管理员误以为立即全局生效

处理：

1. 保存成功提示统一带上“该角色用户需重新登录后生效”。

## 10. 验收标准

满足以下条件则视为完成：

1. 管理员可在前端新增角色。
2. 管理员可勾选并保存角色权限。
3. 系统角色不允许删除，不允许改 `roleCode`。
4. 页面使用真实后端接口，不保留 mock 行为。
5. 重新登录对应账号后，菜单和接口权限变化可见。
